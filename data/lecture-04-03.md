今日は、この「AIが作ったサイト」を「自分のサイト」に変えていきます。全体の色を統一して、スマートフォンでもきれいに見えるか確認して、最後にメタタグという、サイトの裏側の情報も整えていきます。いわば、お店のオープン前の最終チェックですね。では、さっそくはじめましょう。

## 本編

### セクション1: 全体の色統一（8分）

まず最初にやるのは、サイト全体の色を統一する作業です。

今、StackBlitzでカフェサイトのプレビューを見てみてください。ホームページ、メニューページ、ギャラリーページ、それぞれ見比べてみると、微妙に色の雰囲気が違っていませんか。たとえば、ホームページのボタンは青っぽいのに、メニューページの見出しは緑っぽい、なんてことが起きているかもしれません。

これは、ページごとに別々にClaudeにお願いしたからです。AIは一つ一つのリクエストに対してベストを尽くしてくれるんですが、前のページでどんな色を使ったかまでは覚えていないんですね。だから、ページごとに少しずつ違う色を選んでしまうことがあります。

これを直すには、CSS変数という仕組みを使います。CSS変数は今から初めて学ぶ内容ですが、仕組みはシンプルなので安心してください。今から一緒にやっていきましょう。

StackBlitzのファイルツリーから、style.cssを開いてください。このファイルの一番上に、こういうコードを追加します。

```css
:root {
  --color-primary: #5B4A3F;
  --color-secondary: #8B7355;
  --color-accent: #C4956A;
  --color-bg: #FAF6F0;
  --color-text: #333333;
  --color-white: #FFFFFF;
}
```

ルートというのは、CSSの一番上の階層、つまりサイト全体に適用される場所です。ここに色を定義しておくと、サイトのどこからでもこの色を呼び出せるようになります。

今回は、カフェらしい温かみのある色を選んでみました。プライマリーカラー、つまりメインの色は、こげ茶色。セカンダリーカラーは少し明るい茶色。アクセントカラーは、キャラメルのような温かいオレンジ。背景は、ほんのりクリームがかった白。テキストは、真っ黒だときつい印象になるので、少しだけグレーに寄せた黒にしています。

もちろん、この色は好みに合わせて自由に変えてください。カフェのイメージに合わせて、もっと緑っぽくしたり、もっとモダンにグレー系にしたり。たとえば和風のカフェなら深い緑や朱色、北欧風なら淡いブルーやグレー、ポップな雰囲気なら鮮やかなオレンジやイエローなど、色だけでお店の雰囲気がガラッと変わりますよね。

CSS変数のいいところは、ここの値を変えるだけで、サイト全体の色が一括で変わるところなんです。もし変数を使わずに色を直接書いていたら、20か所も30か所も一つずつ変えなきゃいけない。それが、変数を使えばたったの6か所変えるだけで全部反映される。これは本当に便利な仕組みです。

では、この変数を実際に使っていきましょう。style.cssの中で、色を直接指定しているところを探して、変数に置き換えていきます。

たとえば、ナビゲーションバーの背景色が「background-color」のあとにカラーコードで直接書かれていたら、それを変数に置き換えます。

```css
nav {
  background-color: var(--color-primary);
  color: var(--color-white);
}
```

「var」というのは、さっき定義した変数を呼び出す書き方です。「var、かっこ、ハイフンハイフン、変数名、かっこ閉じ」ですね。

同じ要領で、ボタンやリンクの色、見出しの色、フッターの背景色なども、変数に置き換えていきます。全部を一気にやる必要はありません。まずは、目立つところから始めましょう。ナビゲーションバー、ボタン、フッターの3か所を変えるだけで、かなり統一感が出てきます。

ここで一つ、AIにお願いするコツをお見せしますね。style.css全体をコピーして、Claudeに「このCSSで使われている色をすべてCSS変数に置き換えてください。変数はルートで定義済みです」と頼むと、一括で変換してくれます。手作業でひとつひとつ探すより、ずっと効率的です。

ただし、AIが返してきた結果をそのまま貼り付ける前に、ざっと目を通してくださいね。意図しない変更がないか確認する。これが「ディレクター」の仕事です。たとえば、AIがボタンの角丸やマージンまで勝手に変えてしまうことがあります。頼んだのは色の変換だけですから、それ以外の変更は戻した方がいいですよね。

プレビューを確認してみましょう。ホームページ、メニューページ、ギャラリーページ、すべてのページで統一された色になっていますね。最初のバラバラだった状態と比べると、ぐっとプロっぽくなりました。ナビゲーションバーとフッターが同じこげ茶色で、ボタンがキャラメル色のアクセントカラーで揃っていて、全体として「一つのお店のサイト」としてまとまった印象になっています。

ついでに、フォントも統一しておきましょう。Google Fontsを使います。パート3でも一度やりましたね。style.cssの先頭、CSS変数の前に、こういう一行を追加します。

```css
@import url('https://fonts.googleapis.com/css2?family=Noto+Sans+JP:wght@400;700&display=swap');
```

そして、bodyのフォント指定を変更します。

```css
body {
  font-family: 'Noto Sans JP', sans-serif;
  color: var(--color-text);
  background-color: var(--color-bg);
}
```

ノトサンスジェーピーは、日本語がきれいに表示されるフォントです。もちろん他のフォントでも構いません。Google Fontsにはたくさんの無料フォントがありますので、好みのものを選んでみてください。

これで、色もフォントもサイト全体で統一されました。

### セクション2: レスポンシブ確認と修正（10分）

次に、レスポンシブの確認をしていきましょう。レスポンシブというのは、パート3で学んだ「画面サイズに合わせてレイアウトが変わる仕組み」でしたね。

実際のところ、今の世の中、ウェブサイトへのアクセスの半分以上はスマートフォンからなんです。お店のサイトならなおさらですよね。「カフェ 近く」「渋谷 コーヒー おすすめ」なんて検索して、スマートフォンで見る人が圧倒的に多いわけです。もし、そのとき表示されたサイトのレイアウトが崩れていたら、お客さんは「なんだか見づらいな」と思って、すぐに別のお店のサイトに移ってしまいます。だから、スマートフォンでの見た目はとても大切です。

Chromeの開発者ツールを使って確認していきましょう。StackBlitzのプレビュー画面を右クリックして、「検証」を選びます。前に開発者ツールを開いた方法と同じですね。

開発者ツールが開いたら、左上の方にスマートフォンとタブレットのアイコンがあります。これをクリックすると、画面がモバイル表示に切り替わります。上の方にデバイスの選択肢が出てきますので、まずは「iPhone SE」を選んでみましょう。横幅375ピクセルの、小さめのスマートフォンですね。

さて、どうでしょうか。ナビゲーションバーを見てみてください。パソコンではきれいに横並びだったメニューが、スマートフォンだとはみ出していたり、文字が重なっていたりしませんか。

よくある問題を3つ紹介しますね。

1つ目は、今言ったナビゲーションバーです。メニュー項目が横に並びすぎて、画面からはみ出してしまう。これを直すには、スマートフォンサイズのときはメニューを縦並びに変えるか、ハンバーガーメニューにする方法があります。今回は簡単な方法で、メニューを縦並びにしましょう。

style.cssの一番下の方に、メディアクエリのブロックがあるはずです。なければ追加しましょう。

```css
@media (max-width: 768px) {
  nav ul {
    flex-direction: column;
    text-align: center;
  }
}
```

これで、画面幅が768ピクセル以下のとき、つまりタブレットやスマートフォンのとき、ナビゲーションのメニューが縦並びになります。パート3で学んだメディアクエリですね。フレックスディレクションをコラムに変えるだけです。

プレビューを見てみましょう。縦にきれいに並びましたね。

2つ目の問題は、画像ギャラリーです。パソコンでは3列のグリッドで表示していた画像が、スマートフォンだと小さすぎて何が写っているのかわからない。これも、スマートフォンでは1列表示に変えましょう。

```css
@media (max-width: 768px) {
  .gallery-grid {
    grid-template-columns: 1fr;
  }
}
```

グリッドテンプレートコラムズを「ワンエフアール」にすると、1列表示になります。画像が大きく1枚ずつ表示されるので、スマートフォンでも写真がしっかり見えますね。

3つ目は、フォントサイズです。パソコン向けに大きめに設定した見出しが、スマートフォンだと画面いっぱいになってしまうことがあります。

```css
@media (max-width: 768px) {
  h1 {
    font-size: 1.8rem;
  }
  h2 {
    font-size: 1.3rem;
  }
}
```

レムという単位はパート3で学びましたね。こうやって、スマートフォンのときだけ少し小さめのサイズを指定してあげます。

ここで、他にも気になるところがあれば、同じメディアクエリの中に追加していきましょう。パディングが大きすぎるとか、画像が画面からはみ出しているとか。一つずつ確認して、直していく作業です。

確認しながら気づくと思いますが、問題は大抵パターンが決まっています。横幅が固定値で指定されている要素が幅いっぱいに広がりすぎる、テキストが折り返されずはみ出す、ボタンが小さすぎてタップしづらい。こういった問題に出くわすたびに、「スマートフォンのときはこうする」という引き出しが増えていきます。最初は時間がかかりますが、経験を積むうちにだんだん早く直せるようになりますよ。

地味な作業に感じるかもしれませんが、これが実務でもとても大切なんです。AIが最初からある程度レスポンシブに対応してくれていることも多いですが、細かいところは自分の目で確認して調整する。この「最後のひと手間」が、プロっぽい仕上がりにつながります。

実際、プロのウェブ制作の現場でも、レスポンシブの検証と修正に一番時間がかかるという話をよく聞きます。画面サイズのバリエーションは無数にありますからね。でも、今日やっているように「小さいスマートフォン」「タブレット」「パソコン」の3パターンを押さえておけば、ほとんどのデバイスでちゃんと表示されます。

では、デバイスの選択を「iPad Air」に変えてみましょう。タブレットサイズです。こちらも確認してみてください。タブレットの場合は、ギャラリーは2列くらいがちょうどいいかもしれませんね。

```css
@media (min-width: 769px) and (max-width: 1024px) {
  .gallery-grid {
    grid-template-columns: repeat(2, 1fr);
  }
}
```

これで、タブレットのときは2列、スマートフォンのときは1列、パソコンのときは3列になります。画面サイズに合わせて、最適な表示になるわけですね。

デバイスの選択を元に戻して、パソコン表示でも崩れていないか最終確認しましょう。はい、大丈夫ですね。パソコンでもスマートフォンでもタブレットでも、きれいに表示されるサイトになりました。

### セクション3: ページ間ナビゲーションテスト（4分）

レスポンシブの確認ができたところで、次はページ間のナビゲーションをテストしましょう。

カフェサイトは5ページで構成されていますよね。ホーム、メニュー、ギャラリー、アクセス、お問い合わせ。これらのページが、ちゃんとリンクでつながっているか確認します。

StackBlitzのプレビューで、ホームページのナビゲーションバーを見てください。メニュー、ギャラリー、アクセス、お問い合わせ、それぞれクリックしてみましょう。

クリックするたびに、ちゃんと目的のページに移動しますか。移動先のページでも、ナビゲーションバーが表示されていますか。移動先のページから、またホームに戻ってこられますか。

これを全ページで試してください。5ページあるので、少し面倒に感じるかもしれませんが、大切な作業です。実際のお店のサイトで、リンクが切れていたら、お客さんが見たいページにたどりつけませんよね。メニューを見たいのにページが開かない、予約したいのにお問い合わせページに行けない。これではお客さんが離れてしまいます。

もしリンクが切れているところがあったら、HTMLのアンカータグを確認しましょう。パート2で学びましたね。href属性に、リンク先のファイル名が正しく書かれているか。「menu.html」を「memu.html」と打ち間違えていないか。こういった小さなタイプミスが、リンク切れの一番多い原因です。

あと、もう一つ確認してほしいのが、「今いるページ」がナビゲーションバーでわかるようになっているかどうかです。たとえば、メニューページにいるときに、ナビゲーションバーの「メニュー」という文字が太字になっていたり、色が変わっていたりすると、お客さんが「今どこにいるのか」がわかりやすいですよね。

もしまだそうなっていなければ、各ページのナビゲーション部分で、該当するリンクにクラスを追加する方法があります。

```html
<a href="menu.html" class="active">メニュー</a>
```

```css
nav a.active {
  color: var(--color-accent);
  font-weight: bold;
}
```

各ページのHTMLファイルで、そのページに該当するリンクにだけ「active」というクラスをつけます。そして、CSSで「active」クラスがついたリンクのスタイルを変える。これだけで、ナビゲーションがぐっとわかりやすくなります。

### セクション4: メタタグの追加（7分）

さて、ここまでは目に見える部分のカスタマイズでしたが、ここからは「目に見えない」けれど大切な部分を整えていきます。メタタグです。

メタタグというのは、HTMLのヘッドタグの中に書く情報で、ブラウザや検索エンジンに「このページはどんなページですか」を伝えるためのものです。ウェブサイトを訪れたお客さん自身の目には直接見えないんですが、裏側でとても重要な役割を果たしています。

StackBlitzでindex.htmlを開いて、ヘッドタグの中を見てみましょう。今はたぶん、titleタグだけが入っている状態だと思います。

まず、titleタグの内容を整えましょう。

```html
<title>Café Lumière — こだわりのコーヒーとスイーツ</title>
```

このtitleタグは、ブラウザのタブに表示される文字です。今、StackBlitzのプレビューのタブを見てみてください。ここに表示されていますよね。それだけでなく、Googleで検索したときの検索結果にも、このtitleが使われるんです。だから、お店の名前と、何のお店かがわかるような内容にしておくのが大切です。

次に、descriptionメタタグを追加します。

```html
<meta name="description" content="東京・渋谷のカフェ ルミエール。自家焙煎コーヒーと季節のスイーツをご提供しています。Wi-Fi完備、電源あり。">
```

このdescriptionは、Googleの検索結果でタイトルの下に表示される説明文です。お店の場所、特徴、売りを簡潔にまとめます。長すぎると途中で切れてしまうので、120文字くらいを目安にしましょう。

このtitleとdescriptionは、ブラウザのタブだけでなく、Googleの検索結果にも使われます。SEOという言葉で聞いたことがあるかもしれませんが、SEOの詳しい話はパート5でしっかりやります。今はまず正しく設定しておきましょう。

さて、次はOGタグです。OGタグのOGは、オープングラフの略です。

```html
<meta property="og:title" content="Café Lumière — こだわりのコーヒーとスイーツ">
<meta property="og:description" content="東京・渋谷のカフェ ルミエール。自家焙煎コーヒーと季節のスイーツをご提供。">
<meta property="og:type" content="website">
<meta property="og:url" content="https://cafe-lumiere.example.com">
<meta property="og:image" content="https://cafe-lumiere.example.com/images/og-image.jpg">
```

OGタグは何のためにあるかというと、SNSでサイトのURLを共有したときに表示される情報を指定するためのものです。たとえば、LINEで友達にカフェのURLを送ったとき、URLだけがポンと表示されるより、お店の名前と説明と写真が一緒に表示された方が、見た人も「あ、おいしそうなカフェだな」とわかりやすいですよね。

OGタイトル、OGディスクリプションは、titleやdescriptionと同じ内容で大丈夫です。OGタイプは「ウェブサイト」にしておきましょう。OGユーアールエルは、サイトを公開したときの実際のURLを入れます。今はまだ公開前なので、仮の値で構いません。OGイメージは、SNSでシェアされたときに表示される画像のURLです。お店の外観やおすすめメニューの写真がいいですね。

ちなみに、OGイメージは1200かける630ピクセルが推奨サイズです。パート5でサイトを公開するときに、実際の画像を設定しますので、今は仮の値を入れておけば大丈夫です。

ここで、忘れてはいけないメタタグがもう一つあります。ビューポートの設定です。

```html
<meta name="viewport" content="width=device-width, initial-scale=1.0">
```

これは、スマートフォンでサイトを見たときに、ちゃんとスマートフォンの画面幅に合わせて表示してくれる設定です。この一行がないと、スマートフォンなのにパソコンサイズのページがそのまま縮小されて表示されてしまいます。文字が小さすぎて読めない、あの状態ですね。

AIが生成したコードにはすでに入っていることが多いですが、念のため確認しておきましょう。ヘッドタグの中に、このビューポートの一行があるかどうか。なければ追加してください。

最後に、文字エンコーディングの指定も確認しておきましょう。

```html
<meta charset="UTF-8">
```

これも、ほとんどの場合AIが最初から入れてくれていますが、なかったら追加してください。日本語を正しく表示するために必要な設定です。

では、ここまでのメタタグをまとめて確認しましょう。index.htmlのヘッドタグの中身がこうなっていれば大丈夫です。

```html
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Café Lumière — こだわりのコーヒーとスイーツ</title>
  <meta name="description" content="東京・渋谷のカフェ ルミエール。自家焙煎コーヒーと季節のスイーツをご提供しています。Wi-Fi完備、電源あり。">
  <meta property="og:title" content="Café Lumière — こだわりのコーヒーとスイーツ">
  <meta property="og:description" content="東京・渋谷のカフェ ルミエール。自家焙煎コーヒーと季節のスイーツをご提供。">
  <meta property="og:type" content="website">
  <meta property="og:url" content="https://cafe-lumiere.example.com">
  <meta property="og:image" content="https://cafe-lumiere.example.com/images/og-image.jpg">
  <link rel="stylesheet" href="style.css">
</head>
```

これを、ホームページだけでなく、メニューページ、ギャラリーページ、アクセスページ、お問い合わせページのすべてのHTMLファイルに入れていきます。titleとdescriptionは、ページごとに内容を変えるのがベストです。

たとえば、メニューページなら「メニュー | Café Lumière」、ギャラリーページなら「ギャラリー | Café Lumière」というように。ただ、今回は手間もかかりますので、まずはホームページのものをコピーして全ページに入れて、余裕があればページごとに変える、という進め方で大丈夫です。

### セクション5: コードレビューと整理（5分）

さて、いよいよ仕上げの最終段階です。コードの見直しと整理をしていきましょう。

ここでやることは、大きく3つです。

1つ目は、不要なコードの削除です。開発中に試しに書いてそのまま残っているコメントや、使っていないCSSのルールがないか確認します。たとえば、「テスト用の赤い背景」なんていうのが残っていたら、削除しましょう。

2つ目は、コードの見やすさです。インデントがそろっているか、空行が適切に入っているか。これはプレビューの見た目には影響しないんですが、後から自分でコードを見返したときに、整理されていると断然わかりやすいんです。

3つ目は、画像のalt属性の確認です。パート2で学びましたね。imgタグには、altという属性で画像の説明を書きます。たとえば、カフェの外観写真なら「カフェ ルミエールの外観」、ラテアートの写真なら「自家焙煎コーヒーのラテアート」というように。

alt属性は、画像が表示されなかったときの代替テキストとしてだけでなく、視覚に障害のある方がスクリーンリーダーでサイトを閲覧するときにも読み上げられます。つまり、alt属性をきちんと書くことは、アクセシビリティの観点からも大切なんですね。

この3つの作業も、AIにお手伝いしてもらえます。HTMLファイルやCSSファイルの内容をコピーして、Claudeに「このコードをレビューして、改善点があれば教えてください」と頼んでみましょう。AIは、タイプミスや不要なコード、足りないalt属性なども見つけてくれることがあります。

特に、alt属性の確認はAIが得意とする作業です。全部の画像を人間の目で一つずつ確認するのは手間がかかりますが、HTMLをコピーして「imgタグにalt属性が正しく設定されているか確認して」と頼むと、抜けているところをリストアップしてくれます。AIをレビュアーとして活用する、という使い方ですね。

ただし、ここでも大事なのは「AIの提案をそのまま全部受け入れない」ことです。AIは時に、必要以上に複雑なコードを提案することがあります。たとえば、今の段階では不要なJavaScriptのコードを追加しようとしたり、過度にファイルを分割しようとしたり。そういう提案は、今回は見送って大丈夫です。

皆さんはディレクターです。AIの提案を見て、「これは今のサイトに必要だ」と思ったものだけを採用する。その判断力こそが、この講座で身につけたい力ですね。

では、最後にもう一度、サイト全体を通しで見てみましょう。StackBlitzのプレビューで、ホームページから順番に全ページを開いてみてください。

色は統一されていますか。フォントは統一されていますか。ナビゲーションは全ページで機能していますか。スマートフォン表示でも崩れていませんか。

はい、いい感じですね。皆さん、最初にClaudeが作ってくれた状態と、今の状態を比べてみてください。色がバラバラだったのが統一されて、スマートフォンでもきれいに見えるようになって、メタタグも整って。同じコードがベースなのに、ずいぶん印象が変わりましたよね。

ここまでの作業を通して、もう一つ大事なことをお伝えしたいと思います。今日の作業で皆さんがやったこと、色を選ぶ、レイアウトを確認する、リンクをテストする、メタタグを設定する。これらは全部、「自分のサイトを自分の責任で仕上げる」という作業です。AIはたたき台を作ってくれますが、最終的にお客さんに見てもらうサイトの品質を決めるのは、皆さん自身です。そして、ここまでの講義を受けた皆さんには、もうその力があるんです。
