# 講義スクリプト レビューガイド

スクリプト作成後、確定前に実施するレビューの観点と手順をまとめた文書。
レビューの目的は「この1講義が、25講義全体の中で正しい位置に正しい内容で収まっているか」を検証することにある。

---

## 1. カリキュラム準拠チェック

スクリプトが `docs/curriculum-basic.md` の該当講義セクションを忠実に反映しているかを検証する。

### 1-1. 項目網羅

- [ ] カリキュラムの該当講義に列挙されたすべての項目がスクリプトに含まれているか
- [ ] 逆に、カリキュラムに**ない**内容がスクリプトに混入していないか
- [ ] [PPT] / [ライブ] の区分がカリキュラム指定と一致しているか

> **判定方法**: カリキュラムの箇条書きを1行ずつ読み、スクリプト内の対応箇所に下線を引く。対応箇所がない項目は漏れ、対応箇所が見つからないスクリプト内容はカリキュラム外の追加。

### 1-2. パート目標との整合

- [ ] スクリプトの到達目標が、カリキュラムのパート冒頭に記載された「このパートの目標」と矛盾しないか
- [ ] スクリプトの到達目標が、カリキュラムの講義タイトル・内容から自然に導かれる範囲内か
- [ ] 「基礎編の到達目標」（5項目）に反する内容がないか

### 1-3. 形式・時間

- [ ] ヘッダーの「形式」がカリキュラムの [PPT] / [ライブ] / [PPT + ライブ] 指定と一致しているか
- [ ] ナレーション文字数が「想定時間 × 400字/分」の ±10% に収まっているか
- [ ] PPT区間とライブ区間の時間配分がカリキュラムのパート全体比率（例: PART 1 は PPT 60% + ライブ 40%）と大きくずれていないか

---

## 2. 前後講義との連携チェック

スクリプトは孤立した文書ではなく、25講義の連鎖の一環。前後の講義との接続を検証する。

### 2-1. 導入の前回復習

- [ ] 導入部で前回講義の核心を1〜2文で正しく復習しているか（初回講義は除く）
- [ ] 復習で言及する概念・用語が、前回スクリプトで実際に扱った内容と一致しているか
- [ ] まだ確定していない前回スクリプトの内容を前提にしていないか

### 2-2. まとめの次回予告

- [ ] まとめ部で次の講義の内容を具体的に予告しているか
- [ ] 予告内容がカリキュラムの次回講義と一致しているか
- [ ] 次回講義で実際にやらないことを約束していないか

### 2-3. PART切り替え

- [ ] PART最初の講義の場合、ここまでの旅路を短く振り返っているか
- [ ] PART最後の講義の場合、次PARTへの橋渡し文言があるか

---

## 3. 後続講義への影響チェック（最重要）

スクリプト内の発言が、後続講義の内容・構成に意図しない制約を生まないかを検証する。
**これが最も見落としやすく、かつ修正コストが高いポイント。**

### 3-1. 概念の先取り

| チェック項目 | 具体例 |
|---|---|
| 後続講義で初めて扱う概念を、この講義で詳しく説明しすぎていないか | 1-1でFlexboxの仕組みまで踏み込む → 3-5の内容が薄くなる |
| 後続講義で「初登場」として扱う用語を、この講義で先に定義していないか | 1-2で「セマンティックタグ」を定義 → 2-4の初出感が損なわれる |
| 「〜はパートXで詳しくやります」の多用（PART当たり最大2〜3回） | 過度な前振りは受講生の不安を煽る |

- [ ] スクリプト内で初めて登場する専門用語をすべて列挙し、カリキュラム上その用語を本格的に扱う講義を確認。この講義で説明するのが適切か判断したか
- [ ] 「名前だけ出す」「軽く触れる」「詳しく説明する」の3段階のうち、この講義での扱いが適切なレベルか

### 3-2. ツール・手順の言及

| チェック項目 | 具体例 |
|---|---|
| まだ使わないツールの操作手順に踏み込んでいないか | 1-1でCodePenの具体的操作に言及 → 1-4の内容と重複 |
| ツールの紹介レベルがカリキュラム指定と一致しているか | 「実習で使う」vs「紹介のみ」の区分 |
| 後続で変わる可能性のある手順を断言していないか | 「StackBlitzでファイルを…」→ PART 5まで出てこない |

- [ ] スクリプト内で言及するすべてのツール名を列挙し、カリキュラムの「使用ツール」表および各講義内容と照合したか

### 3-3. 約束・期待値の設定

スクリプト中の「宣言」は、後続講義で回収する義務が生じる。

- [ ] 「この講座を終えると〇〇ができるようになります」系の宣言が、カリキュラム全体の到達目標と一致しているか
- [ ] 「パートXで詳しくやります」と言った内容が、実際にカリキュラムの該当パートに存在するか
- [ ] 完成物のイメージ（プロジェクト数、サイトの種類、機能）がカリキュラムと一致しているか
- [ ] 講座の総時間、講義数、パート数などの数字が正確か

### 3-4. 比喩・フレーミングの一貫性

比喩は理解を助けるが、後続講義で別の比喩を使うと混乱を生む。

- [ ] この講義で導入した比喩（例: 家＝HTML/CSS/JS、映画監督＝ディレクター）が、後続講義で使われる比喩と矛盾しないか
- [ ] 繰り返しキャッチフレーズ（「コーダーではなくディレクター」「暗記不要」等）の表現がガイドライン指定の文言と一致しているか
- [ ] 新しい比喩を導入した場合、後続講義でもその比喩を維持できるか（一度限りの比喩は混乱の元）

### 3-5. 難易度カーブの維持

- [ ] この講義の説明深度がパートの難易度設定に合っているか
  - PART 1〜2: 極めてゆっくり。一つも取りこぼさない
  - PART 3: 少しペースを上げるが核心概念は詳しく
  - PART 4: 実習中心。コード理解 > コード暗記
  - PART 5: 実用ガイド。従えばできる安心感
- [ ] 前の講義より急に難しくなっていないか。逆に、後の講義に比べて内容が被りすぎていないか

## 4. ガイドライン準拠チェック

`docs/script-guidelines.md` のルールに従っているかを検証する。

### 4-1. 言語・トーン

- [ ] ナレーション内に英語単語が残っていないか（すべてカタカナ変換済みか）
- [ ] `——`（em ダッシュ）が使われていないか
- [ ] `（）` 括弧による補足挿入がないか
- [ ] 口語の丁寧体で書かれているか（書面体になっていないか）

### 4-2. AIツール関連

- [ ] 実習ツール（Claude、ChatGPT）と紹介のみツールを混同していないか
- [ ] 紹介のみツールに「実際に使ってみましょう」等の操作指示がないか
- [ ] 「プロンプトエンジニアリング」という用語を使っていないか

### 4-3. フォーマット

- [ ] ヘッダー（所属、形式、想定時間、到達目標）が正しいか
- [ ] 「導入 → 本編（セクション分割）→ まとめ」の構造になっているか
- [ ] 【スライド: 〇〇〇】/【AIデモ: 〇〇〇】の表記が正しいか
- [ ] 講師メモが末尾にあるか

### 4-4. 禁止事項

- [ ] スクリプトに存在しない内容を追加していないか
- [ ] 受講生を怖がらせる表現がないか（「これは難しいですが…」→ ✕）
- [ ] 「今後詳しくやります」の多用がないか（PART当たり2〜3回まで）

## 5. レビュー手順

### ステップ1: カリキュラム照合（5分）

1. `docs/curriculum-basic.md` から該当講義のセクションを開く
2. カリキュラムの各箇条書き項目について、スクリプト内の対応箇所を特定する
3. 対応なし＝漏れ、スクリプトにしかない内容＝カリキュラム外追加、として記録する

### ステップ2: 前後接続確認（5分）

1. 前の講義のスクリプト（または前の講義のカリキュラム項目）を開く
2. 導入部の復習内容が前回内容と一致するか確認する
3. まとめ部の次回予告がカリキュラムの次回講義と一致するか確認する

### ステップ3: 後続影響スキャン（10分）

1. スクリプト内の**すべての専門用語**を抽出する
2. 各用語がカリキュラム上どの講義で本格的に扱われるかを確認する
3. この講義での扱いレベル（名前だけ / 軽く触れる / 詳しく説明）が適切か判断する
4. スクリプト内の**すべてのツール名**を抽出し、カリキュラムの使用ツール表と照合する
5. スクリプト内の**数字・宣言**（時間、講義数、完成物、「パートXで」発言）を抽出し、カリキュラムと照合する

### ステップ4: ガイドライン照合（5分）

1. `docs/script-guidelines.md` のチェック項目を流し読みしながらスクリプトを確認する
2. 英語残留、em ダッシュ、括弧補足、禁止表現を検索する

### ステップ5: 判定

- **LGTM**: 全チェック項目がクリア
- **要修正**: 問題箇所を具体的に指摘し、修正案を添える
- **要再設計**: カリキュラムとの構造的不整合がある場合。スクリプトの構成自体を見直す

### ステップ6: レビュー結果の保存

レビューが終わったら、結果を必ず `tmp/review/` に保存する。

- 保存先: `tmp/review/`
- ファイル名形式: `<元スクリプトファイル名>.review.md`
- 言語: レビュー結果は日本語で書く
- トラック: `data/` 直下のディレクトリ名を使う
  - 例: `data/lecture-01.md` → `tmp/review/lecture-01.review.md`
- 元スクリプトファイル名: 拡張子 `.md` を除いたベース名を使う
- 同じ講義を再レビューした場合は、同じファイル名で上書きして最新版を保つ

レビュー結果ファイルには最低限、以下を含める。

- 対象スクリプトのパス
- 判定（LGTM / 要修正 / 要再設計）
- 指摘事項。重要度の高い順に並べる
- 必要なら修正方針または修正案
- レビュー実施日

### ステップ7: 修正完了後のコミットとプッシュ

レビュー結果に基づく修正が完了したら、変更内容をコミットし、リモートにプッシュする。

- コミットメッセージは韓国語で書く
- レビュー対応に関係する変更だけをステージングする
- 他の未関連変更は勝手に含めない
- プッシュまで完了して、レビュー対応を完了扱いにする
