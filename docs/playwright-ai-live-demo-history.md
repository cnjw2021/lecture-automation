# AI 라이브 데모 씬 자동화 — 접근 이력과 현행 아키텍처

이 문서는 Claude · ChatGPT 같은 AI 서비스를 브라우저에서 실제로 조작해 녹화하는 **AI 라이브 데모 씬**을 자동화하기 위해 본 프로젝트가 거쳐온 시행착오와, 그로부터 수렴한 **현행 파이프라인 설계**를 기록한다.

> **이 문서의 목적**
>
> 미래의 작업자(사람·AI 어시스턴트 모두)가
> 1. **이미 실패·폐기된 접근 방식을 재시도하지 않도록** 한다.
> 2. 현재 설계가 왜 이렇게 복잡한지를 **최초 설계 의도와 함께** 이해할 수 있도록 한다.
> 3. 남아 있는 **깨지기 쉬운 포인트(fragility)** 를 명시적으로 관리한다.

---

## 1. 문제의 본질 — 왜 어려운가

AI 라이브 데모 씬은 다음 네 가지가 **동시에** 성립해야 한다.

1. **녹화 타이밍이 가변적**: Claude 응답 생성 시간은 매번 다름 (보통 5~60초, 때로는 180초까지).
2. **나레이션 길이는 고정**: 나레이션은 대본 기반으로 확정되며 이후 길이가 바뀌지 않음.
3. **나레이션과 화면은 특정 시점에서 싱크가 맞아야 함**: "出てきましたね" 가 들릴 때 화면에 응답이 보여야 자연스러움.
4. **AI 서비스 UI 는 통제 불능**: DOM 구조, 클래스명, 스트리밍 상태 표시가 공급자에 의해 언제든 바뀔 수 있음.

이 네 제약이 "단순히 녹화해서 이어붙이면 되지 않느냐" 는 직관을 깨뜨린다. 녹화 길이가 매번 다르기 때문에 **고정된 나레이션 길이에 맞추는 보정 메커니즘**이 반드시 필요하다.

---

## 2. 폐기된 접근 방식

### ❌ 접근 1: 단일 씬 올인원 녹화 (초기)

- **방식**: 하나의 Playwright 씬이 `goto → 입력 → 송신 → 응답 대기 → 스크롤 → 결과 관찰` 전체를 녹화.
- **실패 이유**:
  - 응답 대기 시간 가변성 때문에 씬 총 길이가 매번 달라짐 → 나레이션과 어긋남.
  - `wait ms` 를 수동 튜닝해도 다음 녹화에서 응답이 빨라지면 공백 시간 발생, 느려지면 스크립트가 잘림.
  - 총 씬 길이가 60초를 넘으면 재녹화 실패 시 코스트가 매우 큼.
- **폐기일**: 2026-03 ~ 2026-04 초 (정확 시점 불명).

> TTS 전략(마스터 오디오, 청크 TTS 등) 의 시행착오 이력은 본 문서 범위가 아니며 `docs/tts-history.md` 에 별도 기록되어 있다.

## 3. 현행 접근 — 씬 분할 + urlFromScene + 역방향 싱크

**채택일**: 2026-04-17 ~ 2026-04-18
**관련 커밋**: `467cd61 feat: AI 라이브 데모 씬 분할 + urlFromScene 참조 메커니즘`

### 3.1 핵심 아이디어

AI 라이브 데모를 **송신 씬**과 **결과 확인 씬**으로 분리한다.

| 씬 | 역할 | 녹화 시간 |
|---|---|---|
| **송신 씬** (예: 28) | Claude 입력·송신까지. `press Enter` 후 URL 이 `/chat/{uuid}` 로 전이될 때까지 대기. | 고정 (나레이션 길이) |
| **[중간 슬라이드]** (예: 28.5) | Artifact 설명 등 Remotion 슬라이드. 실제로는 이 시간 동안 Claude 가 응답을 생성 중. | 고정 |
| **결과 확인 씬** (예: 29) | 송신 씬에서 얻은 conversationUrl 을 다시 열고, 응답 완료를 확인하고, Artifact 프리뷰를 스크롤. | **가변** → 역방향 싱크로 오디오에 무음 삽입 |

슬라이드 씬이 중간에 삽입되어 **송신씬↔결과 확인씬 사이에 발생하는 응답 생성 시간**을 흡수한다 (수강생 관점에서 단절 없이 Artifact 설명이 이어짐).

> ⚠️ **중요한 제약 — 슬라이드가 흡수하지 않는 시간**
>
> 슬라이드 씬이 흡수하는 것은 "씬 28 끝 ~ 씬 29 시작" 사이에 Claude 가 응답을 생성하는 시간뿐이다.
> **씬 29 자체의 `goto` + `wait_for_claude_ready` + `click` + 패널 슬라이드인 시간은 씬 29 녹화 내부의 시간으로 남는다.**
>
> 즉, 결과 확인 씬 녹화 타임라인은 다음과 같다:
> ```
> 0s       goto urlFromScene:28 (2~3s 로딩)
> ~3s      wait_for_claude_ready (2s ~ 180s 가변)
> ~Ns      wait (DOM 정착)
> ~Ns      click Artifact 카드
> ~Ns      wait (패널 슬라이드인)
> ~Ns      이제서야 프리뷰가 화면에 보이기 시작
> ```
>
> 이 때 결과 확인 씬의 **첫 나레이션 구절이 "프리뷰가 보이는 시점" 을 전제**로 작성돼 있다면 (예: "出てきましたね", "나왔네요"), 시작부 싱크가 **물리적으로 맞지 않는다**.
> 반드시 3.2 (4) 의 **syncPoint 앞쪽 고정** 기법을 적용하거나, 나레이션을 로딩 시간에 맞춰 재작성해야 한다.

### 3.2 주요 메커니즘 4종

#### (1) `urlFromScene` — 씬 간 상태 전달

결과 확인 씬은 송신 씬이 만든 Claude 대화 URL 을 재사용해야 한다.

```json
{"cmd": "goto", "urlFromScene": 28, "note": "씬 28 이 만든 대화 URL 재사용"}
```

Provider 는 씬 N 의 녹화 매니페스트(`scene-N.manifest.json`) 에서 `conversationUrl` 필드를 읽어 실제 URL 로 치환한다.

#### (2) `wait_for_claude_ready` — 응답 완료 폴링

결과 확인 씬 진입 시점에 Claude 응답이 완료됐는지 불확실하므로, **10초 간격으로 폴링**하며 최대 180초 대기한다.

```json
{"cmd": "wait_for_claude_ready", "timeout": 180000}
```

완료 판정 조건 (세 신호 조합):
- `[data-is-streaming="false"]` 존재
- 빈 인사말("本日はどのようなお手伝いができますか？") 부재
- 사용자 메시지 + 어시스턴트 응답 DOM 존재

대기 중 CLI 에 한국어 5단계 회전 메시지 출력 (동영상엔 포함되지 않음).

#### (3) Stage 0 사전 녹화 — TTS 이전에 라이브 데모 수행

가변 녹화 시간을 가진 씬은 **TTS 생성보다 먼저** 녹화해야 녹화 매니페스트의 타임스탬프를 기준으로 오디오 길이를 산출할 수 있다.

파이프라인 순서:
```
0) 라이브 데모 씬 사전 녹화 (filterLiveDemo=true)
1) TTS 생성
1.5) 오디오 미리듣기 머지
1.7a) 역방향 싱크 (비디오 → 오디오 무음 삽입)
1.7b) 순방향 싱크 (오디오 → 비디오 wait 재계산)
2) 스크린샷
3) 일반 시각자료 녹화 (filterLiveDemo=false)
4) 씬 클립 렌더링
5) concat
```

#### (4) 역방향 싱크 — 녹화 길이에 오디오를 맞춤

결과 확인 씬의 녹화 매니페스트에서 특정 액션(예: `scroll`) 의 시작 시각을 가져와, 대응 나레이션 구절(`syncPoints.phrase`) 발화 시점에 **무음을 삽입**해 WAV 총 길이를 녹화 길이와 정합시킨다.

```json
"syncPoints": [{"actionIndex": 8, "phrase": "スクロールしていくと"}]
```

> ⚠️ **설계 제약 — 첫 나레이션 구절이 시작 0ms 가 아닌 액션에 대응하는 경우**
>
> `syncPoints` 는 **명시된 구절만 보정**한다. syncPoint 가 없는 구절은 자동으로 맞지 않는다.
> 따라서 나레이션 첫 구절이 녹화 도중 특정 시점(예: Artifact 패널 오픈 완료) 을 전제로 한다면, **해당 시점을 가리키는 syncPoint 를 앞쪽에 반드시 추가**해야 한다.
>
> 예:
> ```json
> "syncPoints": [
>   {"actionIndex": 4, "phrase": "出てきましたね", "target": "end"},  // 패널 슬라이드인 wait 종료 = 프리뷰 보이기 시작
>   {"actionIndex": 8, "phrase": "スクロールしていくと"}
> ]
> ```
> 이렇게 하면 역방향 싱크가 "出てきましたね" 앞에 무음을 삽입해 프리뷰가 실제로 보이는 시점에 정렬한다.
>
> **체크리스트**: 결과 확인 씬을 설계할 때 다음을 반드시 확인할 것.
> 1. 첫 나레이션 구절이 0ms 에 시작해도 화면이 자연스러운가? (예: "では見てみましょう" 같은 중립적 도입)
> 2. 아니라면 해당 구절을 시점이 명확한 액션(click, wait 종료 등) 과 syncPoint 로 묶었는가?
> 3. syncPoint 로 묶어 앞에 무음이 생길 경우, 그 무음 구간(로딩 애니메이션만 보이는 구간) 의 길이가 수강생이 견딜 만한가? 너무 길면 나레이션을 도입 문장으로 채워야 한다.

## 4. 현 구현의 깨지기 쉬운 포인트

### 4.1 Claude DOM 셀렉터 (High Risk)

- **Artifact 카드 클릭**: `button[aria-label*='アーティファクトを開く']`.
  - Claude UI 업데이트 시 버튼 구조 · 라벨 변경 가능.
  - 2026-04-18 실측: `button:has-text('コード'):has-text('HTML')` 로 10초 타임아웃 발생.
    버튼 자체는 `absolute inset-0` 오버레이라 내부 텍스트 없음 — "コード"/"HTML" 은 형제 div 라벨. `aria-label` 은 `"{title}。アーティファクトを開く。"` 형태라 접근성 계약에 묶여 더 안정적.
- **빈 인사말 감지 문자열**: `本日はどのようなお手伝いができますか？`.
  - 시간대·계정 설정·버전에 따라 문구가 달라질 수 있음.
  - 실패 시 `wait_for_claude_ready` 가 영원히 대기.

### 4.2 라이브 데모 씬 판정 로직 분산 (Medium Risk)

`wait_for` 또는 `wait_for_claude_ready` 를 기준으로 라이브 데모 씬을 판정하는 코드가 **4곳** 에 분산됨. 새 cmd 추가 시 전부 갱신해야 함.

| 파일 | 용도 |
|---|---|
| `RunAutomationPipelineUseCase.hasLiveDemoScenes()` | Stage 0 실행 gate |
| `RunAutomationPipelineUseCase.hasForwardSyncableScenes()` | 1.7b gate (라이브 데모는 제외) |
| `RecordVisualUseCase.computeLiveDemoSceneIds()` | 씬 단위 필터 (+ urlFromScene 의존 체인 확장) |
| `ReverseSyncPlaywrightUseCase` | 역방향 싱크 대상 판정 |
| `SyncPlaywrightUseCase` | 씬 단위 skip (순방향 싱크 대상 아님) |

**개선 여지**: 도메인 유틸 함수로 통합 (`isLiveDemoScene(scene)`).

### 4.3 Stage 0/3 순서와 `urlFromScene` 의존성 (Medium Risk, 2026-04-18 해결)

**발생한 사고**: 씬 29 (라이브 데모) 가 `urlFromScene:28` 을 참조하는데, 씬 28 은 `wait_for` 계열 cmd 가 없어 라이브 데모로 판정되지 않음 → Stage 0 에서 씬 29 만 먼저 녹화 → 씬 28 의 이전 실행 매니페스트(stale URL) 를 읽어버림.

**해결**: `computeLiveDemoSceneIds()` 가 라이브 데모 씬의 `urlFromScene:N` 참조를 따라가 씬 N 도 라이브 데모 집합에 포함하도록 함. Stage 0 에서 씬 순서대로 처리되므로 송신 씬이 먼저, 결과 확인 씬이 나중에 녹화됨.

**남은 주의**: `regen-scene SCENE=29` 처럼 결과 확인 씬만 재생성하면 씬 28 은 target 이 아니므로 재녹화되지 않고 기존 매니페스트를 사용. 이는 의도된 동작이지만, 사용자가 "29만 다시 찍겠다" 고 할 때 송신 씬도 함께 지정해야 함을 명심할 것.

### 4.4 녹화 매니페스트 overwrite 타이밍 (Low Risk)

`regen-scene` Makefile 타겟은 webm 은 삭제하지만 `.manifest.json` 은 남겨둔다. Stage 0 에서 재녹화 시 Provider 가 매니페스트를 overwrite 하므로 실질적 문제는 없음. 단, Provider 가 매니페스트 저장 **이전** 에 실패하면 stale 매니페스트가 남는 점에 유의.

### 4.5 결과 확인 씬 시작부 싱크 (High Risk, 2026-04-18 발견)

**발생한 사고**: 씬 29 나레이션이 "出てきましたね (나왔네요)" 로 시작하지만, 녹화 타임라인은 `goto` → `wait_for_claude_ready` → `click` → 패널 슬라이드인 순서라 실제 프리뷰가 보이는 시점은 씬 시작 후 10~20초 이후. `syncPoints` 에 "스크롤" 관련 1개만 있었고 시작부 phrase 가 없었기 때문에 역방향 싱크가 시작부에 무음을 삽입하지 못하고 싱크가 완전히 깨짐.

**핵심 원인**: `syncPoints` 는 **명시된 구절만** 보정한다는 설계 특성을 간과. 첫 나레이션 구절이 녹화 도중 특정 순간을 전제하면 그 순간을 가리키는 syncPoint 가 반드시 필요함에도, 라이브 데모 씬 설계 시 이 점이 체크리스트에 없었음.

**대응 방침** (3.2 (4) 경고 블록에 정리): 결과 확인 씬 설계 시 반드시 다음 3개 질문을 통과해야 함.
1. 첫 나레이션 구절이 0ms 시점에 자연스러운가?
2. 아니라면 시점을 가리키는 syncPoint 가 앞에 있는가?
3. syncPoint 로 생성되는 무음 구간(로딩만 보이는 구간) 이 견딜 만한 길이인가?

**남은 근본적 한계**: 아무리 정교하게 syncPoint 를 배치해도, 결과 확인 씬 시작부에는 **화면은 비어 있지만 나레이션이 진행되는 구간** 혹은 **화면은 로딩 중이지만 나레이션 없는 구간** 중 하나가 반드시 생긴다. 이를 완전히 없애려면 "Playwright 브라우저 컨텍스트를 씬 간 공유" 라는 별도 메커니즘이 필요하며, 이는 현재 아키텍처 범위를 벗어남.

### 4.6 scroll 대상 패널 내부에 커서를 먼저 두지 않으면 엉뚱한 영역이 스크롤됨 (High Risk, 2026-04-18 발견)

**발생한 사고**: 씬 29 결과 확인 구간에서 `mouse_move [720, 400]` → `scroll` 순서로 작성했는데, `[720, 400]` 은 1920×1080 뷰포트 기준 **좌측 채팅 영역** 좌표였다. `page.mouse.wheel(0, deltaY)` 는 현재 커서가 놓인 스크롤 컨테이너만 스크롤하므로, Artifact 프리뷰 대신 대화 이력이 스크롤됐다. 나레이션은 "プレビューをスクロールしていくと…" 을 설명하고 있어 시청자 입장에서 싱크가 완전히 깨짐.

**핵심 원인**: Claude 는 좌측 채팅과 우측 Artifact 패널이 **각각 독립된 스크롤 컨테이너** 라는 점을 간과. `scroll` 액션 명세에 커서 위치 의존성이 명시돼 있지 않았고, json-conversion-rules.md 의 예시 자체가 채팅 좌표를 쓰고 있어 그대로 복제되기 쉬웠음.

**대응 방침**:
1. `scroll` 액션 직전에 반드시 `mouse_move` 로 **대상 스크롤 컨테이너 내부** 좌표를 찍는다.
2. Claude.ai 1920×1080 기준 주요 좌표는 `docs/json-conversion-rules.md` 의 Claude 뷰포트 좌표 참고표를 사용한다 (Artifact 패널 중앙 ≈ `[1500, 500]`).
3. 셀렉터 깨짐처럼 로그에 에러가 뜨지 않고, 녹화 영상을 실제로 봐야 드러나는 **조용한 버그** 이므로, 라이브 데모 씬 수정 후에는 반드시 webm 을 열어 스크롤 대상을 육안 확인할 것.

### 4.7 Claude Artifact 비결정성 — 프롬프트가 출력 구조를 좌우 (High Risk, 2026-04-19 발견)

**발생한 사고 3건** (lecture-01-03 씬 28·29 작업 중):

1. **Artifact 미생성 (인라인 코드블록으로 출력)**: 프롬프트 `"マッサージサロンのホームページを作ってください。メニュー表、アクセス情報、お問い合わせボタンを含めてください。"` 로 요청했을 때, Claude 가 대화창 내 인라인 코드블록으로 HTML 을 출력하고 Artifact 패널을 열지 않음. 씬 29 전제("右側にプレビューが表示されています") 가 깨짐.
   - **해결**: 프롬프트 말미에 `"インラインのコードブロックではなく、アーティファクトとして作成してください。"` 추가. 이 2문이 있으면 Claude 는 streaming 중 Artifact 패널을 자동으로 연다.

2. **Artifact 자동 오픈 후 click 타임아웃**: 위 명시 지시로 Artifact 가 자동 오픈되는 상황에서, 기존 설계(§3.3) 의 `click button[aria-label*='アーティファクトを開く']` 가 10초 타임아웃. 패널이 이미 열려 있어 버튼 DOM 이 존재하지 않기 때문.
   - **해결**: 명시 지시를 쓰는 경우 click 을 제거. `wait_for streaming='false'` 후 `wait 5000` (프리뷰 로드) + `mouse_move [1500, 500]` 만으로 프리뷰 상호작용 가능.

3. **나레이션과 Claude 출력 구조 불일치**: 프롬프트가 느슨한 상태에서 나레이션이 "上にサロンの名前が大きく" "メニューがカード型できれいに並んでいて" "色も統一感があって" 등 구체적 레이아웃·스타일을 단언. 실제 Claude 출력은 실행마다 다름 (Haiku/Sonnet/Opus 간 차이, 같은 모델 내에서도 생성마다 구조 변동).
   - **해결**: 프롬프트에서 구조·순서·톤을 고정. 예:
     ```
     ページは上部にヘッダーとナビゲーション、コンセプトの紹介、施術メニュー
     をカード型で料金付き、アクセス情報と営業時間、お問い合わせボタン、とい
     う順で構成してください。落ち着いた統一感のある配色にしてください。
     ```
   - 프롬프트가 구조를 명시해도 **세부 디자인(정확한 색상 값, 폰트, 마이크로카피)은 여전히 비결정적** 이므로, 나레이션에서도 세부까지 단언하지 않는다.

**핵심 원인**: 동영상 나레이션은 "출력 화면" 을 설명하므로, 일반 사용자 프롬프트와 달리 **프롬프트 자체가 출력 구조의 계약** 이어야 함. 이 관점이 최초 설계에 없었다.

**대응 방침** (docs/json-conversion-rules.md "프롬프트 설계 원칙" 에 정리):
1. 프롬프트에 `"インラインのコードブロックではなく、アーティファクトとして作成してください。"` 필수.
2. 나레이션이 언급하는 모든 레이아웃·순서 요소를 프롬프트에서 명시 고정.
3. 프롬프트에서 고정하지 않은 속성(구체 색상, 폰트 등) 은 나레이션에서 단언하지 않는다.

### 4.8 Artifact 프리뷰 스크롤 총 픽셀 예산 부족 (Medium Risk, 2026-04-19 발견)

**발생한 사고**: 씬 29 에서 `scroll deltaY 300` × 3회 = 900px 총 스크롤. 전형적인 사로나이트 HTML 출력(2500~3500px) 의 상단 1/3 정도만 보여주고 나레이션이 끝남. "アクセス情報もあります" "お問い合わせボタンもあります" 나레이션이 화면에 도달하지 못한 상태에서 종료.

**핵심 원인**: json-conversion-rules.md 의 예시 자체가 `deltaY 300 × 3` 이었고, 예상 높이 대비 스크롤 예산 산정 가이드가 없었음.

**대응 방침** (docs/json-conversion-rules.md "스크롤 총 픽셀 예산" 표 신설):
- 1페이지 랜딩(2500~3500px) → **6 × deltaY 400 = 2400px** + 최종 5s hold
- 각 스크롤 사이 wait 3~4s (나레이션 읽는 속도 정합)
- 첫 녹화 후 webm 을 열어 프리뷰 하단까지 도달했는지 육안 확인. 못 도달하면 deltaY 또는 횟수 증가.

### 4.9 디버그 사이클 비용 (High Friction)

- 1회 파이프라인 실행: 오디오 생성(~30s) + 녹화(~수십초/씬) + 렌더(~분) ≒ **5~10분 이상**.
- 실패가 녹화 단계에서 발생하면 즉시 피드백 가능하지만, 렌더링 단계에서 문제 발견 시 사이클이 길어짐.
- Claude API 호출이 포함되므로 **비용도 발생** (Anthropic credit 소모).

## 5. 시작부 싱크 — 패턴별 분석 (100% 보장 없음)

4.5 에서 서술한 "결과 확인 씬 시작부 싱크 불일치" 문제에 대해, 현재 아키텍처 범위 내에서 생각할 수 있는 모든 대응 패턴과 그 한계를 정리한다. **이 중 어느 것도 100% 를 보장하지 못한다.** 100% 는 오직 브라우저 컨텍스트 씬 간 공유(5-D) 를 구현해야만 달성 가능하다.

### 5.1 패턴 비교표

| 패턴 | 개요 | 보장 범위 | 한계 | 현재 상태 |
|---|---|---|---|---|
| **P-A** 시작부 syncPoint 고정 | 결과 확인 씬 첫 구절을 "패널 오픈 완료" 액션에 syncPoint 로 묶어, 역방향 싱크가 앞에 무음 삽입 | 이론적으로 항상 싱크 일치 | `wait_for_claude_ready` 가 길 때 씬 시작부에 **로딩만 보이는 긴 무음 구간** 이 생김 | 메커니즘은 구현 완료, 나레이션이 시작부에 "출てきましたね" 류 표현을 쓰는 경우에 적용 |
| **P-B** 나레이션을 로딩 서술로 도입 | 결과 확인 씬 첫 구절을 "では、結果を見てみましょう" 같은 시점 중립 표현으로 작성 → "出てきましたね" 는 중간으로 밀고 syncPoint 로 묶음 | 대부분의 평균 케이스에서 자연스러움 | `wait_for_claude_ready` 가 도입 나레이션보다 길면 결국 무음 구간 발생. 반대로 응답이 매우 빠르면 도입 나레이션이 불필요하게 느껴짐 | 스크립트 작성 단계의 규칙으로 고정 필요 (`docs/json-conversion-rules.md`). 현재 미반영 |
| **P-C** 송신·슬라이드 길이로 응답 완료 확정 | 송신 씬 + 슬라이드 씬 총 길이를 충분히 길게 해서 결과 확인 씬 진입 시 Claude 응답이 거의 확실히 완료되도록 설계 → `wait_for_claude_ready` 소요를 평균 1~3초로 축소 | 평균 케이스의 대기 시간 단축 | Claude 속도 변동(캐시, 서버 부하) 에 따라 슬라이드가 과잉 또는 부족. **예측이 원리적으로 불가능** | 스크립트 작성 시 슬라이드 분량 가이드 필요. 현재 미반영 |
| **P-A + P-B 조합** | 두 패턴 동시 적용 | P-A 의 "무음 너무 김" 케이스를 P-B 의 도입 나레이션이 덮음 | 극단적으로 응답이 느리면 여전히 무음 발생. 극단적으로 빠르면 도입 나레이션 잉여 | **현시점의 best-effort 추천안** |
| **P-D** 브라우저 컨텍스트 씬 간 공유 | 송신 씬의 브라우저 세션을 결과 확인 씬이 이어받음 → 결과 확인 씬 진입 시 이미 패널 오픈 상태 | 이론상 100% 싱크 가능 | `IVisualProvider` 인터페이스 · Playwright 컨텍스트 라이프사이클 · Stage 0 녹화 분할 규칙 전면 재설계 필요. 구현 비용 매우 큼 | 미시도. 현재 아키텍처 범위 밖 |

### 5.2 패턴 선택 기준

- **나레이션이 이미 확정된 경우**: P-A 우선. 시작부 syncPoint 를 추가하고 무음 구간 길이가 허용 범위인지 검증. 허용 불가능하면 나레이션 부분 수정으로 P-A + P-B.
- **나레이션을 아직 쓰지 않은 경우**: 처음부터 P-B 패턴으로 작성. 이후 P-A 를 겹쳐 적용.
- **반복 재녹화를 피하고 싶은 경우**: 5-A (webm 커밋) 고려. 그러면 P-A/B/C 고민 자체가 불필요해짐.
- **장기적 해결**: P-D 가 유일한 근본 해결책. 다만 착수 결정은 "라이브 데모 씬이 몇 개 이상 되면 투자 가치가 있는가" 로 판단해야 함.

### 5.3 솔직한 자기 평가

현행 접근(씬 분할 + urlFromScene + 역방향 싱크) 은 **시작부 싱크 문제가 근본 해결되지 않은 상태의 불완전 설계**다. 2026-04-18 현재, P-A + P-B 조합으로 대부분의 강의에서 "견딜 만한" 결과를 낼 수 있다고 예상하지만, **엣지 케이스에서 수 초~수십 초의 어색한 구간이 생길 가능성은 상시 존재**한다.

향후 라이브 데모 씬이 5개 이상으로 늘어나거나, 엣지 케이스 사고가 반복 발생하면 P-D 또는 5-A 로 전환하는 것을 권장한다.

## 6. 아직 시도하지 않은 대안

현행 방식이 복잡함 때문에 언젠가 재검토할 가능성이 있는 방향들. 재시도 전에 반드시 "왜 시도하는지" 를 아래 실패 이력에 비추어 재확인할 것.

### 5-A. 라이브 데모 결과물 webm 커밋

- 라이브 데모 씬을 **한 번만** 수동으로 완벽하게 녹화해 webm/manifest 를 저장소에 커밋.
- 이후 파이프라인은 해당 씬의 녹화를 스킵하고 커밋된 webm 을 사용.
- **장점**: Claude 호출 반복 제거, 셀렉터 깨짐 사고에서 해방, 비용 0.
- **단점**: 나레이션 수정 시 재녹화 필요, 저장소 용량 증가, 녹화 시점의 UI 영구 고정.

### 5-B. 녹화 캐싱 (로컬)

- 녹화 webm 이 존재하면 재사용, 없으면 한 번만 생성.
- 현행 `regen-scene` 이 강제로 webm 을 지우는 것과 상반되므로 별도 플래그 필요.
- **장점**: 일상 빌드 시간 단축.
- **단점**: 캐시 무효화 정책 복잡, 실제로 "업데이트 필요" 한 경우 수동 삭제 번거로움.

### 5-C. 셀렉터 대신 스크롤 + 시간 기반 대기

- Artifact 카드를 클릭해 패널을 여는 대신, Claude 가 응답 완료 시 자동으로 여는 Artifact 패널을 기다리기만 함.
- 셀렉터 의존도 감소 → 깨짐 위험 감소.
- **시도 전 검증 필요**: 신규 대화 URL 을 재로드했을 때 Artifact 패널이 자동으로 열리는가? (열리지 않는다면 클릭 불가피).

## 6. 작업 시 지침

### 6.1 라이브 데모 씬을 만들 때

1. 송신 씬과 결과 확인 씬을 분리한다. 사이에 Remotion 슬라이드를 넣어 송신↔결과 확인 사이의 응답 생성 시간을 흡수한다. (슬라이드가 **결과 확인 씬 내부의 로딩 시간은 흡수하지 못함**. 4.5 참조)
2. 결과 확인 씬에는 반드시 `storageState`, `wait_for_claude_ready`, `syncPoints` 세 가지를 설정.
3. `urlFromScene` 으로 송신 씬을 참조.
4. 클릭 셀렉터는 **언어 중립적** 인 것을 우선 (예: `[aria-label]`, 안정적 클래스명). `has-text` 기반은 최후의 수단.
5. **결과 확인 씬 시작부 싱크 체크리스트 (필수, 3.2 (4) · 4.5 참조)**:
   - 첫 나레이션 구절이 0ms 에 시작해도 화면이 자연스러운가? 아니라면 시점을 가리키는 syncPoint 를 앞쪽에 추가.
   - syncPoint 로 생기는 무음 구간(로딩만 보이는 구간) 이 수강생이 견딜 만한가? 너무 길면 나레이션에 도입 문장 추가.
   - 이 체크가 끝나기 전에는 실제 녹화·렌더링 파이프라인을 돌리지 말 것 (디버그 사이클 비용이 높음).
6. **`scroll` 직전 커서 배치 체크 (필수, 4.6 참조)**: Artifact 프리뷰 등 페이지의 특정 패널을 스크롤하려면 `scroll` 직전에 `mouse_move` 로 **대상 패널 내부** 좌표에 커서를 둔다. Claude 1920×1080 기준 우측 Artifact 패널 중앙은 `[1500, 500]`. 채팅 영역 좌표(x<1040) 로 두면 대화 이력이 스크롤된다 (로그엔 에러 없음, 영상으로만 드러남).
7. **프롬프트 설계 체크 (필수, 4.7 참조)**: `type.key` 에 아래 두 조건을 반드시 충족시킨다.
   - 말미에 `"インラインのコードブロックではなく、アーティファクトとして作成してください。"` 포함 (Artifact 자동 오픈 결정성 확보).
   - 나레이션이 언급하는 레이아웃·순서·톤을 프롬프트에서 구조적으로 고정 (느슨한 프롬프트 → Claude 출력 비결정 → 나레이션 불일치). 고정하지 않은 속성은 나레이션에서도 단언 금지.
8. **Artifact 패널 click 제거 (필수, 4.7 참조)**: 7번의 명시 지시를 사용하면 Claude 가 streaming 중 패널을 자동으로 연다. `click button[aria-label*='アーティファクトを開く']` 액션을 추가하면 버튼 DOM 부재로 10초 타임아웃이 발생하므로 **넣지 말 것**. `wait_for streaming='false'` → `wait 5000` (프리뷰 로드) 로 충분.
9. **스크롤 총 픽셀 예산 확보 (필수, 4.8 참조)**: 결과 확인 씬은 Artifact 프리뷰 하단까지 보여주고 끝나야 한다. 1페이지 랜딩 기준 **6 × deltaY 400 = 2400px** + 최종 5s hold 를 기본형으로 잡고, 첫 녹화 후 webm 으로 하단 도달 여부를 육안 확인해 조정.

### 6.2 재녹화할 때

- 라이브 데모 씬(예: 29) 을 재녹화하려면 반드시 송신 씬(예: 28) 을 **함께** `SCENE` 에 포함.
- `make regen-scene LECTURE=xxx SCENE='28 29'` 와 같이.

### 6.3 실패 진단 순서

1. 파이프라인 로그에서 `0단계` 가 출력됐는지 확인. 없으면 `hasLiveDemoScenes()` 판정 실패 가능성.
2. Stage 0 에서 송신 씬과 결과 확인 씬이 **순서대로** 녹화됐는지 확인. 송신 씬이 누락되면 `computeLiveDemoSceneIds()` 의존 체인 확장이 안 된 것.
3. `⚠️ Action 'click' 실패 (건너뜀): Timeout 10000ms exceeded` 로그가 있으면 DOM 셀렉터 깨짐. 녹화 webm 을 열어 실제 화면 상태 확인.
4. `[ReverseSync]` 로그에서 무음 삽입 양이 음수면 오디오가 비디오보다 김 → 나레이션이 너무 길거나 녹화가 너무 짧음.

## 8. P-D(공유 세션) 부분 재생성의 본질적 제약

### 8.1 Claude 응답 비결정성

shared session에서 씬 28은 `press Enter`로 Claude에 프롬프트를 전송한다.
`make regen-scene SCENE='29'`처럼 씬 29만 재생성하면:

- 파이프라인이 세션을 새로 열고 씬 28을 **replayOnly** 모드로 실행
- replayOnly는 `press Enter`를 포함한 모든 액션을 실제로 실행
- 즉, **매 partial rerun마다 Claude에 새 대화가 생성**된다

결과:
- **Claude API 비용** 누적 (매 재실행 = 새 요청)
- **씬 29 스크린샷이 비결정적** — Claude 응답 내용이 실행마다 다를 수 있음
- full run과 partial rerun의 최종 영상은 **내용이 다를 수 있다**

이것은 버그가 아니라 외부 서비스(AI) 의존 구조의 본질적 특성이다.

### 8.2 실용적 지침

- **씬 29만 재생성하는 것은 피한다.** 항상 `SCENE='28 29'`로 함께 재생성한다.
- 이렇게 하면 씬 28 캡처가 재생성되고(새 Claude 응답), 씬 29가 그 상태를 이어받는다.
- 단, 씬 28 캡처가 재생성되므로 비용은 동일하게 발생한다.
- 스크린샷 품질이 중요한 경우에는 재생성 후 결과를 반드시 확인한다.
