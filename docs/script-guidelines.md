# プロジェクト指示書：AI時代のWeb制作マスター — 講義スクリプト作成

## プロジェクト概要

일본에서 비전공자 대상으로 홈페이지 제작 방법을 알려주는 동영상 강의(약 20시간, 42강)를 만들고 있다. 개발자 양성이 목표가 아니라, AI 시대에 간단한 홈페이지를 누구나 만들 수 있을 정도의 기본 지식과 기초 체력을 갖추도록 하는 것이 목표다. 첨부한 커리큘럼을 기반으로 각 강의별 일본어 스크립트를 작성한다.

---

## 対象受講生ペルソナ

- 일본에 사는 20~50대 비전공자 (사무직, 자영업자, 부업 희망자 등)
- 프로그래밍 경험 없음. "HTML이 뭔지" 정도만 들어본 수준
- PC 기본 조작은 가능 (파일 저장, 브라우저 사용 등)
- AI 도구(ChatGPT, Claude 등)를 약간 써봤거나 관심이 있음
- 동기: 자기 가게/사무소 홈페이지, 자기소개 사이트, 부업(Web制作) 등
- 업종 예시: 미용실, 네일살롱, 정체원, 세무사 사무소, 개인 교실(요리·영어·요가), 소규모 EC 등

## AIツール使用ガイドライン

### 実習に使うツール（受講生が直接操作）
- **Claude (claude.ai)** — メインのコード生成ツール。無料プランでHTML/CSS生成可能
- **ChatGPT** — サブのコード生成ツール。Claudeとの比較実習（4-1）で使用

この2つは無料でHTML/CSSを直接生成でき、受講生が実習中に課金・クレジット切れで詰まることがない。

### 紹介のみのツール（デモまたはスライドで存在を知らせる）
以下のツールは**講師がデモを見せる程度**にとどめる。受講生に直接操作させない。

この区分はスクリプト上でも明確に見えるようにする。該当講義では、ナレーションまたはスライド内で必ず「紹介のみ」「この講義では実習に使わない」と一度は明言する。

#### AIツールマップ（1-4）で紹介するツール
| ツール | 紹介時に言及するポイント |
|---|---|
| **v0.dev** | UI生成に強力だが、出力がReact＋Tailwindベースで純粋なHTML/CSSとは異なる。無料クレジットに上限あり（月$5相当） |
| **Bolt.new** | フルスタックアプリを素早く作れるが、トークン消費が大きく無料プランに制約あり |
| **Canva** | 一般ユーザーにも親しみやすく、レイアウトやビジュアルの方向性を素早く確認しやすい。紹介のみにとどめ、実習では使わない |
| **Figma Make** | デザインやプロトタイプのたたき台作成に便利。ただし本講座の実習フロー（HTML/CSSを読んで直す）とは目的が異なるため紹介のみにとどめる |

#### AI Web制作ツールガイド（4-1）で紹介するツール
VS Codeを学んだ後の時点で初めて意味を持つ開発者向けツール。PART 1時点では受講生にとって文脈がないため、4-1まで言及しない。
| ツール | 紹介時に言及するポイント |
|---|---|
| **Cursor** | AI内蔵エディタで強力だが、VS Codeを学んだ後に追加ツールを導入すると混乱。無料プランに制約あり（月2,000回補完） |
| **GitHub Copilot** | コード補完。有料（月$10）。名前と用途のみ紹介 |

### スクリプト作成時のツール関連原則
- カリキュラムに明記されたツールを該当講義でそのまま使用。任意でツールを追加・交換しない
- 実習ツール（Claude、ChatGPT）と紹介ツールを混同しない
- 紹介ツールを扱う際は、メリットとともに制約事項（無料上限、出力形式の違いなど）を正直に言及する
- 専用デザインツール（Canva、Figma Makeなど）は**コア実習の対象ではない**。デザインの話をするときも、基本は参考サイト見学・画像選定・CSS調整で進める
- 例外として、PART 5のOG画像作成ではCanvaの簡単な実演を入れてよい。ただし「画像作成の補助」であり、Webデザイン実習の中心ではないと明確にする
- 「プロンプトエンジニアリング」という用語は使わない。代わりに「AIへの指示の出し方」「AIへの頼み方」など平易な表現を使う

## 実習プロジェクトのフレーミング

カリキュラムの3つのプロジェクトは以下のようにフレーミングする：

### プロジェクト1: プロフィール・名刺サイト（난이도 ★★）
- カリキュラム上の「個人ポートフォリオサイト」を**自己紹介・名刺サイト**としてリフレーミング
- 「名刺代わりに使えるWebページ」— 自営業者、フリーランス、会社員誰にでも必要
- 技術的内容（ヒーロー、自己紹介、スキル、連絡先）はカリキュラムそのまま維持
- スクリプトで「ポートフォリオ」という言葉の代わりに「プロフィールサイト」「自己紹介サイト」「Web名刺」などを使用

### プロジェクト2: 店舗・サロン紹介サイト（난이도 ★★★）
- カリキュラム上の「カフェ/レストランサイト」を**汎用の店舗紹介サイト**としてリフレーミング
- 実習例はカフェを使うが、受講生が自分の業種にすぐ応用できることを強調
- スクリプトで適切なタイミングに応用例を入れる：
  - 「今回はカフェを例にしていますが、美容室なら施術メニュー、整体院なら施術コース、教室なら開講スケジュールに置き換えるだけです」
- 技術的内容（複数ページ、ギャラリー、Googleマップ、営業時間、問い合わせフォーム）はカリキュラムそのまま維持

### プロジェクト3: サービスランディングページ（난이도 ★★★）
- カリキュラムそのまま維持
- マーケティング観点のページ設計を学ぶプロジェクト

## スクリプト作成ルール

### 言語およびトーン
- **日本語**で作成。自然な口語の丁寧体（〜ですね、〜しましょう、〜ですよね）
- 講師がカメラ/画面を見ながら話す口語スクリプト — 書面体ではない
- 親しみやすいが軽くなりすぎない。硬すぎる教科書体 ✕、軽すぎるYouTuber体 ✕
- トーン参考レファレンス: ドットインストール、Progate公式YouTubeの親切な解説トーン
- 専門用語が初登場する際は必ずわかりやすい比喩や一行説明をつける
- 英語技術用語は日本語読みを併記（例：「Flexbox（フレックスボックス）」）
- ただし基本用語（HTML、CSS、URL等）は英語のまま表記可。詳細は `docs/json-conversion-rules.md` の英語表記ルール参照

### 分量基準
- **1分 ≒ 約400字（日本語）** を基準に算出
- 各講義時間はカリキュラムに明記されている。その時間に合った分量を守る
  - 例：20分講義 → 約8,000字 / 30分講義 → 約12,000字
- この `1分 ≒ 約400字` は、**講義全体の総文字数を決める最優先ルール** とする
- ライブ講義、PPT+ライブ講義、AIデモを含む講義でも、**講義全体の原稿量は原則としてこの基準で確保する**
- 画面操作・タイピング・待ち時間がある講義では、その時間を埋めるナレーションを別途入れて調整する。`ライブだから少なくてよい` とは解釈しない

### スクリプトフォーマット

各スクリプトは以下の構造に従う：

```
# 講義 XX: [タイトル]
- 所属: PART X
- 形式: PPT / ライブ / AIデモ / PPT+ライブ / PPT+AIデモ
- 想定時間: XX分
- 到達目標: [이 강의를 듣고 나면 수강생이 할 수 있는 것 1~2줄]

---

## 導入（1~2分）

[前回の復習 or フック。수강생의 관심을 끄는 질문/상황 제시]

## 本編

### セクション1: [소제목]
[본문 스크립트]

### セクション2: [소제목]
[본문 스크립트]

（필요에 따라 セクション 추가）

## まとめ（1~2分）

[이번 강의 핵심 3줄 요약 + 다음 강의 예고]

---
### 講師メモ
- [화면에 보여줄 것, 전환 타이밍, 주의사항 등을 간결하게]
```

### ライブコーディング区間の表記法
- コードは \`\`\`html / \`\`\`css などのコードブロックで表記
- コードの前後に講師が「今これをタイピングします」式のナレーションを入れる
- 単にコードを並べず、1行または1ブロックごとになぜこう書くか説明を入れる
- 例：
  ```
  では、ここにh1タグを書いてみましょう。

  ```html
  <h1>こんにちは、世界！</h1>
  ```

  h1は「一番大きな見出し」という意味です。新聞の一面の大見出しをイメージしてください。
  ```

### PPT区間の表記法
- `【スライド: ○○○】` でスライド切り替えタイミングを表示
- スライドに載せる核心内容を括弧内に簡潔に記載
- 例：
  ```
  【スライド: ブラウザとサーバーの関係図】

  皆さんがブラウザにURLを入力すると、裏側ではこういうことが起きています。
  まず、ブラウザが「このURLのページをください」とサーバーにリクエストを送ります...
  ```

### AIライブデモ区間の表記法

AIツール（Claude、ChatGPTなど）の画面を操作するデモ区間のルール。Playwright 씬으로 자동 녹화되므로 실제 화면 동작과 나레이션이 정확히 일치해야 한다.

#### 表記
- `【AIデモ: ○○○】` でデモ開始を明示（例: `【AIデモ: ClaudeにHTMLを生成させる】`）
- デモ中の操作手順を `→` で順序表記：
  ```
  → プロンプト入力 → 送信 → 応答待ち → 結果確認
  ```

#### 応答待ち時間の処理
AIツールの応答には数秒〜数十秒かかる。**無音の空白時間を作らない**。待ち時間には以下いずれかのナレーションを入れる：

1. **今起きていることの説明**: 「エーアイがコードを書いている間、少し待ちましょう」
2. **機能の補足説明**: 直前に使った機能や、今見えているUI要素の説明
3. **後の講義への前振り**: 「この機能についてはパート○で詳しく触れますので、今はこういうことができるんだな、くらいで大丈夫です」
4. **受講生への語りかけ**: 「実際にやってみると、こんな感じで進んでいくんですね」

❌ 応答を即座に受け取ったかのような省略は禁止。「では送信しましょう。はい、出てきましたね」のように待ち時間をゼロに描写してはいけない。

#### 画面描写の正確性
ナレーションで画面に見えるものを説明する際、**自動録画で実際に表示される内容**と一致させる：

- AIが生成したコードを説明する場合: 実際の画面表示形態を確認して記述する
  - ✅ Claudeの場合: 「アーティファクトとしてプレビューが表示されています」
  - ❌ 実際にはArtifactプレビューなのに「コードがずらっと表示されています」
- ブラウザで直接レンダリングされる場合: 「ブラウザに表示されましたね」
- ✅ 目に見える要素を具体的に: 「見出しがあって、説明文があって、メニューのカードがあって…」
- ❌ 画面にない要素の描写、または推測に基づく描写

#### AIツール固有のUI機能への言及
AIツールの特徴的な機能（Claudeのアーティファクト、ChatGPTのCanvas等）が画面に表示される場合：

- **初出の講義で1〜2文の簡潔な説明を入れる**: 「クロードにはアーティファクトという便利な機能があって、生成したコードのプレビューをその場で見せてくれるんです」
- **詳細解説は該当カリキュラムに委ねる**: 「この機能についてはパート○で詳しく触れます」
- **同一講義内で同じ説明を繰り返さない**: 初回言及時のみ説明、以降は名前だけで参照

#### 分量上の注意
- AI応答待ちがあるデモ区間でも、待ち時間ナレーション、観察ポイントの言語化、受講生への声かけを入れて、**講義全体の総文字数は必ず通常基準の中で確保する**
- 応答待ちの間も無音にせず、目安として10〜30秒分（50〜150字）程度のナレーションを用意する

#### 例
```
【AIデモ: ClaudeにカフェのHTMLを生成させる】

では、送信してみましょう。
エーアイがコードを書いている間、少し待ちましょう。

クロードにはアーティファクトという便利な機能があって、
生成したコードのプレビューをその場で見せてくれるんです。
この機能についてはパート4で詳しく触れますので、
今はこういうことができるんだな、くらいで大丈夫です。

出てきましたね。では、もう少し大きく見てみましょう。
見出しがあって、説明文があって、メニューのカードがあって、
アクセス情報があって、お問い合わせのボタンもあります。

数年前なら、これをゼロから作るにはかなりの時間がかかっていました。
それが今は、ほんの短時間で、ひとまず形になるんです。
```

## 일관성 유지 규칙

### 講義間の連携
- 毎回の導入部で前回講義の核心を1〜2文で復習
- 毎回の最後に次の講義の予告（具体的に何をするか）
- PART切り替え時はここまでの全体の旅路を短く振り返る

### 繰り返し使うキャッチフレーズ / 概念
- 「AIに正確に指示し、結果を判断できる人」— 講座の核心目標
- 「暗記する必要はありません。何を検索すべきかを知ればいいんです」
- 「コーダーではなくディレクター」— AI時代の役割再定義
- これらの表現はPARTが変わるとき、またAI実習パートで自然に繰り返す

### 用語統一
- エディタ → 基本エディタは「VS Code」。PART 4-1のCursor紹介区間でのみCursorを別途言及
- ブラウザ → 基本的にChrome前提
- 「ホームページ」と「Webサイト」→ 受講生目線で「ホームページ」をメインに使いつつ、業界用語では「Webサイト」と呼ぶことをPART 1で一度説明
- AIツール → カリキュラムに明記されたツールを該当講義でそのまま使用。複数ツールが言及される場合（例：2-7「Claude/ChatGPT」）カリキュラム表記に従い、任意に一本化しない

### 難易度カーブ
- PART 1〜2: 極めてゆっくり。一つも取りこぼさないという感覚
- PART 3: 少しペースを上げるが、核心概念（Flexbox、レスポンシブ）は引き続き詳しく
- PART 4: 「一緒に作りながら学ぶ」実習中心。コード理解 > コード暗記
- PART 5: 実用的ガイド。従えばできるという安心感

## スクリプト作成時にやってはいけないこと

- ❌ 日本語として不自然な直訳体（韓国語や英語の語順が透けて見える文章）
- ❌ 1講義内でカリキュラムにない内容で範囲を逸脱する
- ❌ 過度に長い雑談や自分語り（講師個人エピソードは最小限に）
- ❌ コードを説明なしに一気に見せる
- ❌ 「今後詳しくやります」を多用する（最大PART当たり2〜3回まで）
- ❌ 受講生を怖がらせる表現（「これは難しいですが...」→ 代わりに「最初は不思議に感じるかもしれませんが、すぐ慣れます」）
- ❌ 紹介専用ツール（v0.dev、Cursorなど）を受講生に直接操作させる指示
- ❌ 「プロンプトエンジニアリング」など不必要に専門的な用語の使用
- ❌ AIライブデモで応答待ち時間を省略した描写（「送信しましょう。はい、出てきました」のように瞬時に結果が出たかのような表現）
- ❌ AIツール画面の実際の表示と異なるナレーション（例: Artifactプレビューが出ているのに「コードが表示されている」と説明）
- ❌ AIツール固有機能（Artifact、Canvas等）を説明なしに言及 — 初出時に必ず1〜2文の説明を入れる

## 作業フロー

1. 1講義単位でリクエストする（例：「講義01のスクリプトを作成して」）
2. 完成したスクリプトを確認 → 修正要求があれば反映
3. 確定後に次の講義へ進む
4. 完成したスクリプトはマークダウンファイルで提供する

### PRタイトル
```
강의 X-X 스크립트 작성: 日本語の講義名
